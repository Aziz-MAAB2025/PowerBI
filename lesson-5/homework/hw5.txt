1. What is a primary key in a table?

A primary key is a column (or set of columns) that uniquely identifies each record in a table.
‚úÖ Example:

In Customer, the CustomerID uniquely identifies each customer.

In Products, the ProductID is the primary key.

2. Name the two types of table relationships in Power BI.

One-to-Many (1:*): One record in one table relates to many records in another.

Many-to-Many (:): Records in both tables can have multiple matches.

4. What is a "star schema"?

A star schema is a database design where:

A central Fact Table (e.g., Sales) stores quantitative data (measures).

Surrounding Dimension Tables (e.g., Customer, Products, Date) store descriptive attributes.

Visually, it looks like a star ‚≠ê ‚Äî the fact table is in the center, connected to dimensions.

7. Why is ProductID in Sales.csv a foreign key?

Because it refers to (or ‚Äúpoints to‚Äù) a record in another table ‚Äî the Products table.

Products.ProductID = primary key

Sales.ProductID = foreign key linking back to it.

This ensures each product in Sales exists in the Products table.

9. Explain why a star schema improves performance.

Simpler joins: Only one level of relationships between fact and dimensions.

Faster queries: Power BI‚Äôs VertiPaq engine compresses columns better.

Reduces ambiguity: No circular relationships.

Enables DAX optimization: Functions like RELATED and SUMMARIZE work efficiently.

10. Add a new column TotalSales in Sales (Quantity * Price from Products).

Use a calculated column (DAX):

TotalSales = Sales[Quantity] * RELATED(Products[Price])


‚úÖ RELATED fetches the corresponding price from the Products table via the existing relationship.

11. Optimize a model with circular relationships ‚Äî how would you resolve it?

Remove unnecessary links (keep one clear path).

Use DAX functions (e.g., USERELATIONSHIP()) instead of extra relationships.

Introduce bridge tables to break loops.
‚úÖ Example: If both Sales ‚Üí Products and Sales ‚Üí Categories cause a loop, merge the category info into Products.

13. Handle a many-to-many relationship between Customers and Products.

Since many customers can buy many products:

Create a bridge table like Sales that contains unique combinations of CustomerID and ProductID.

Use it to connect both Customers and Products indirectly.

‚úÖ Avoids ambiguous relationships and allows aggregation through the bridge.

Use bidirectional filtering only when both tables need to filter each other for analysis.
‚úÖ Example:
